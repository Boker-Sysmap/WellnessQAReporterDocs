Test Runs

Esta página define como as Test Runs devem ser criadas e utilizadas no Qase, estipulando as regras de nomenclatura e procedimento obrigatórias para garantir rastreabilidade, consistência e cálculo correto dos KPIs utilizados pelo WellnessQAReporter.

Objetivo

Estabelecer diretrizes claras para criação e execução das Test Runs, garantindo que cada execução esteja corretamente vinculada ao Test Plan correspondente, e siga o padrão oficial de identificadores.

1. O que é uma Test Run?

A Test Run é a execução prática de um Test Plan. Ela registra:

  • quem executou
  • quando executou
  • o status de cada caso
  • evidências
  • defeitos encontrados

Por que isso importa?

Os KPIs utilizam as Test Runs para calcular:

  • percentual real de execução
  • taxas de Passed / Failed / Blocked / Retest
  • cobertura da release
  • defeitos associados aos casos

2. É proibido criar Test Run sem Test Plan

Toda Test Run deve obrigatoriamente ser criada a partir de um Test Plan.

Motivos:

  • O KPI “Total de Testes Planejados” depende exclusivamente do Test Plan.
  • Sem Test Plan não é possível calcular cobertura planejada x executada.
  • O Test Plan é a única definição oficial do escopo da release.

✔ Test Run SEMPRE tem Test Plan

❌ Nunca crie uma Test Run direta, manual ou solta

3. Identificador oficial (nome da Run)

A Test Run deve usar exatamente o mesmo identificador do Test Plan correspondente.

O formato oficial do identificador é:

${version}_${environment}_${platform}_${language}_${testType}_${date}

Onde:

  • version: formato X.Y.Z (ex.: 3.2.1)
  • environment: PROD, DEV, STAGE
  • platform: IOS, ANDROID, WEB
  • language: PT, EN, ES
  • testType: funcional_manual, regressao, smoke, etc.
  • date: AAAAMMDD (data real da execução)

Exemplos válidos:

3.2.1_PROD_IOS_PT_[FUNCIONAL_MANUAL]_20251117

3.2.1_DEV_ANDROID_EN_MANUAL_20251011

Regras importantes:

  • Campos com espaços, acentos ou caracteres especiais devem estar entre colchetes.
  • Os campos são sempre separados por underline (_).
  • Usar somente caracteres MAIÚSCULOS.
  • Test Plan e Test Run devem ter exatamente o mesmo nome.
  • O nome é usado pelo WellnessQAReporter para identificar a release.

4. Regras obrigatórias da Test Run

  • Deve ser criada a partir de um Test Plan.
  • Deve ser nova (nunca reaproveitada).
  • Deve seguir o padrão oficial de nomenclatura.
  • Deve registrar status em 100% dos casos.
  • Deve relacionar defeitos corretamente.
  • Deve ter executor definido.
  • Deve ter data real (AAAAMMDD) de execução.

5. Erros proibidos

Os itens abaixo são proibidos e invalidam KPIs:

  • ❌ Criar Test Run sem Test Plan
  • ❌ Reaproveitar uma run antiga
  • ❌ Misturar plataformas na mesma run
  • ❌ Criar Test Run com nome fora do padrão
  • ❌ Criar run depois do término da execução
  • ❌ Deixar casos sem status
  • ❌ Renomear runs manualmente de forma inconsistente

6. Execução Correta

O QA deve garantir que:

  • o ambiente está estável antes de executar
  • segue exatamente os passos do Test Case
  • registra evidências para resultados não-passados
  • classifica corretamente o status
  • abre defeitos para qualquer falha real
  • revisa a execução antes de encerrar a run

7. Fechamento da Test Run

Uma run só é considerada válida quando:

  • 100% dos casos possuem status
  • todos os defeitos foram registrados
  • não há pendências sem justificativa
  • a equipe revisou e aprovou a execução

8. Impacto nos KPIs

As Test Runs influenciam diretamente:

  • Escopo planejado x executado
  • Cobertura real
  • Distribuição Passed / Failed / Blocked / Retest
  • Defeitos por funcionalidade
  • Severidade de falhas
  • Qualidade por plataforma

Conclusão

As Test Runs são fundamentais para garantir KPIs precisos e rastreabilidade real da qualidade da versão. O uso padronizado e disciplinado é obrigatório para todos os QAs envolvidos no projeto.
