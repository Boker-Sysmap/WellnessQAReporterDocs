Test Runs

Esta página define como as Test Runs (Execuções de Teste) devem ser criadas e utilizadas no Qase para garantir que o WellnessQAReporter consiga interpretar corretamente o progresso da release, calcular cobertura real e relacionar resultados aos defeitos.

Objetivo:

Estabelecer regras claras e obrigatórias para criação e uso das Test Runs, evitando falhas que impactam diretamente os KPIs e a rastreabilidade das execuções.

1. O que é uma Test Run?

A Test Run é a execução prática de um Test Plan.
Ela registra:

  • quem executou
  • quando executou
  • qual caso foi aprovado ou falhou
  • quais defeitos foram identificados

Por que isso importa?

O Reporter utiliza as Test Runs para gerar:

  • porcentagem de execução
  • taxa de sucesso e falha
  • métricas reais de cobertura
  • defeitos associados aos casos

2. Uma Test Run por Plataforma

Para cada release, devem ser criados:

  • 1 Test Run para iOS
  • 1 Test Run para Android

Nunca misture execuções das duas plataformas em uma única Test Run.

3. Nomenclatura Oficial

A Test Run deve seguir o padrão:

Test Run – Release YYYY-MM – Plataforma

Exemplos:

  • Run – Release 2025-03 – iOS
  • Run – Release 2025-03 – Android

Por que isso importa?

O WellnessQAReporter utiliza esse padrão para identificar corretamente:

  • qual plataforma pertence a cada execução
  • distribuição dos KPIs por iOS x Android

4. Regras obrigatórias da Run

Toda run deve:

  • ser criada a partir de um Test Plan
  • ser nova (nunca reaproveitada)
  • estar associada à release correta
  • ter responsável (executor) definido
  • ter data real de execução
  • registrar resultados de todos os Test Cases

5. Erros proibidos

Os erros abaixo comprometem totalmente os KPIs:

  • ❌ Reaproveitar uma run antiga
  • ❌ Criar a run diretamente sem um plan
  • ❌ Deixar casos sem status
  • ❌ Misturar plataformas na mesma run
  • ❌ Criar run após finalizado o ciclo
  • ❌ Executar apenas parte dos casos e abandonar a run
  • ❌ Repetir a mesma run com nome diferente

6. Execução Correta

Para garantir execução correta, o QA deve:

  • iniciar execução com ambiente estável
  • seguir os passos descritos no case
  • registrar evidências sempre que necessário
  • classificar corretamente o resultado
  • registrar defeitos para qualquer falha
  • revisar execução antes de encerrar a Test Run

7. Fechamento da Test Run

Uma Test Run só pode ser considerada válida quando:

  • 100% dos casos têm status
  • Todos os defeitos foram registrados
  • Não há casos pendentes sem justificativa
  • O QA líder revisou e aprovou a execução

8. Impacto nos KPIs

A forma como a Test Run é executada afeta diretamente:

  • KPIs de Escopo
  • Total de casos planejados vs executados
  • KPIs de Qualidade
  • Percentual de Passed / Failed / Blocked / Skipped
  • KPIs de Defeitos
  • Falhas mapeadas por funcionalidade
  • Severidade
  • Plataforma afetada

9. Boas Práticas

Revisar Test Runs diariamente durante o ciclo.

Registrar comentários claros em cada caso não aprovado.

Anexar evidências objetivas.

Encerrar Test Run apenas após revisão da equipe.

Criar Test Runs com antecedência, antes da execução começar.

Conclusão:

AS Test Runs são um dos pilares principais do WellnessQAReporter.

Quando bem executados, garantem KPIs confiáveis, rastreabilidade completa e visibilidade real da qualidade da release.

O uso padronizado é obrigatório para todos os QAs envolvidos.
