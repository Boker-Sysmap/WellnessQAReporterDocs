KPI – Escopo: Número total de casos planejados para a release

1. Objetivo do KPI

Indicar quantos casos de teste fazem parte da release, ou seja, qual é o escopo planejado do ciclo.

Esse KPI é a base para outros indicadores, como:

  • Cobertura planejada x executada
  • Percentual de execução
  • Percentual de casos bloqueados
  • Percentual de falhas por release
  • Tendência de retrabalho e regressão

2. Regras de Negócio

2.1 RN01 – Definição de Escopo da Release

O Escopo de uma Release é composto por todos os Test Cases associados a um Test Plan que esteja relacionado à Release, conforme configurado no Qase.

Definição formal:
Escopo = soma de todos os casos de teste contidos em todos os Test Plans vinculados à Release, desconsiderando duplicidades.

2.2 RN02 – Deduplicação

Se um mesmo Test Case aparecer em mais de um Test Plan dentro da mesma release, ele deve ser contado apenas uma vez no KPI de escopo.

2.3 RN03 – Versionamento (Android/iOS) e Idiomas

Cenário recomendado (melhor prática)

Utilizar um único Test Case e controlar variações via tags de plataforma e idioma, por exemplo:

  • Plataforma: android, ios
  • Idioma: pt, en, es

Neste modelo, o case conta uma única vez no escopo da release. As diferenças são controladas por tags ou pelo contexto de execução.

Cenário alternativo (cliente exige separação)

Se o mesmo cenário for cadastrado como cases diferentes (por exemplo, um case para Android e outro para iOS), cada case passa a ser uma entidade independente no Qase e, portanto, conta separadamente no escopo.

Exemplos de nomes de cases:

  • [Android] Pagamentos – Pix – Sucesso
  • [iOS] Pagamentos – Pix – Sucesso

2.4 RN04 – Casos fora do Test Plan

Casos de teste que não estão vinculados a nenhum Test Plan, mesmo existindo no projeto, não fazem parte do escopo da release.

2.5 RN05 – Casos removidos do Plan

Se um QA remover um case manualmente de um Test Plan durante o ciclo, esse case deixa de ser contabilizado no escopo da release a partir da remoção.

3. Cálculo do KPI no WellnessQAReporter

3.1 Fonte de Dados (API Qase)

Para calcular esse KPI, o WellnessQAReporter utiliza os seguintes recursos da API do Qase (ou equivalentes):

  • /plan – lista de Test Plans
  • /plan/{id} – detalhes do plano e seus cases
  • /case – metadados dos Test Cases
  • /suite – opcional, para agrupamentos por suíte
  • Metadados de release – via campo de release do Plan, tags ou padronização de nomes

3.2 Lógica de cálculo

  1. Identificar a release alvo (por nome, versão e/ou datas).
  2. Localizar todos os Test Plans vinculados a essa release.
  3. Coletar todos os IDs de Test Case contidos nesses planos.
  4. Inserir os IDs em uma estrutura que elimine duplicidade (por exemplo, um set).
  5. O total de elementos únicos nesse conjunto é o Escopo Total da Release.

4. Configuração necessária no Qase

4.1 Criação de Releases formais

É obrigatório usar o módulo de Releases do Qase para que o KPI funcione corretamente.

Cada release deve conter, no mínimo:

  • Nome padronizado (ex.: Fully – 2025.11.15 (R24))
  • Datas de início e fim
  • Status adequado (ativo, em andamento, concluído etc.)

Sem a criação formal de releases, o Reporter não consegue consolidar o escopo corretamente.

4.2 Associação de Test Plans à Release

Em cada Test Plan, é necessário preencher o campo de Release.

Passos gerais:

  1. Acessar o Test Plan.
  2. Editar o plano.
  3. Selecionar a Release correspondente no campo apropriado.
  4. Salvar o plano.

Planos não associados a nenhuma release não serão considerados no cálculo deste KPI.

4.3 Evitar execuções avulsas

Para manter o KPI íntegro, os testes devem ser executados a partir de Runs vinculadas a Test Plans.

Execuções “soltas” (sem Plan) devem ser desencorajadas por processo, pois dificultam rastreabilidade e distorcem os indicadores.

4.4 Padronização de tags

Recomenda-se uma lista oficial de tags para controlar plataforma, idioma e outras dimensões relevantes, por exemplo:

  • android
  • ios
  • pt
  • en
  • es

Isso permite, futuramente, desdobrar o KPI de escopo por plataforma ou idioma, se necessário.

4.5 Padrão de nomenclatura de suítes e cases

Exemplo de padrão sugerido:

  • Suítes: [Módulo] – [Submódulo]
  • Cases: [Funcionalidade] – [Cenário] – [Especificidade]

Exemplos:

  • Suite: Pagamentos – Pix
  • Case: Pagamentos – Pix – Sucesso

Isso facilita o agrupamento por funcionalidade, a leitura dos relatórios e comparações entre projetos (Fully x Chubb).

4.6 Organização Android/iOS

Opção A – Melhor prática (case único com tags)

  • Case único representando o cenário de negócio.
  • Uso de tags para diferenciar plataforma (android/ios).
  • Execuções diferenciadas por ambiente na Run, se necessário.

Opção B – Estrutura separada (cenário atual)

  • Cases duplicados por plataforma.
  • Cada case conta individualmente no escopo da release (um para Android, outro para iOS).
  • Uso de prefixos no nome para facilitar identificação, por exemplo:
    • [Android] Pagamentos – Pix – Sucesso
    • [iOS] Pagamentos – Pix – Sucesso

5. Riscos e pontos de atenção

Problema de configuração / processo Risco para o KPI Ação recomendada
Execuções de teste sem uso de Test Plan Perda de rastreabilidade e escopo incompleto Orientar equipe a sempre usar Test Plans vinculados à release
Planos não associados a nenhuma release Release aparece com escopo zero ou subdimensionado Treinar equipe para sempre selecionar a release no cadastro do plano
Casos duplicados sem necessidade Escopo inflado e indicadores distorcidos Revisar estrutura de cases e adotar uso de tags ao invés de duplicação
Nomenclatura inconsistente Dificuldade para agrupar métricas por funcionalidade Definir e comunicar um padrão oficial de nomenclatura de suites e cases
Tags usadas de forma diferente entre membros da equipe Métricas por idioma/plataforma imprecisas ou quebradas Criar e divulgar uma lista oficial de tags permitidas e exemplos de uso

6. Resumo executivo

  • O escopo da release é definido pelos Test Cases presentes nos Test Plans vinculados à release.
  • O WellnessQAReporter deduplica automaticamente cases repetidos em mais de um plano.
  • É obrigatório:
    • Usar releases formais no Qase;
    • Associar todos os Test Plans à release correspondente;
    • Evitar execuções avulsas fora de planos.
  • Padronização de nomenclaturas e tags é essencial para relatórios limpos e comparáveis.
