Registro de Resultados

Registro de Resultados (Passed, Failed, Blocked, Skipped)

Esta página define como os resultados devem ser registrados dentro das Runs no Qase.

O preenchimento correto dos resultados é essencial para que o WellnessQAReporter calcule KPIs precisos, rastreie falhas, identifique padrões e gere relatórios confiáveis.

Objetivo:

Garantir que todos os casos executados em uma Test Run recebam um status coerente, com justificativas e evidências adequadas.

1. Importância do Registro de Resultados

O resultado de cada caso de teste impacta diretamente:

  • KPIs de execução (quantidade e porcentagem)
  • KPIs de qualidade (taxas de sucesso e falha)
  • Visão por funcionalidade
  • Detecção e classificação de defeitos
  • Rastreabilidade do ciclo de testes

Resultados incompletos, incorretos ou sem justificativa comprometem todo o relatório.

2. Tipos de Resultados e Quando Utilizar

2.1 Passed

Use Passed quando:

  • o caso de teste executou exatamente como esperado
  • não houve erro, travamento ou comportamento estranho
  • o fluxo corresponde à documentação do case

Nunca utilize Passed se houver qualquer anomalia observada.

2.2 Failed

Use Failed quando:

  • o caso não obteve o resultado esperado
  • ocorreu erro funcional ou de interface
  • o fluxo foi quebrado
  • comportamento inesperado foi encontrado

Toda falha deve gerar um defeito.

Nenhuma run deve conter Failed sem issue (defect) associada.

2.3 Blocked

Use Blocked quando:

  • o app não permite iniciar o teste
  • há impedimento externo (ex.: login indisponível, servidor fora do ar)
  • a funcionalidade está inacessível

Importante: Blocked não significa falha funcional — significa que o teste não pôde ser executado.

2.4 Skipped

Use Skipped quando:

  • o caso não se aplica à plataforma
  • o cenário é inválido para aquela versão
  • há dependência quebrada no próprio case

Skipped deve ser exceção, nunca regra.

3. Regras Obrigatórias de Preenchimento

  • Nenhum caso pode ficar sem status.
  • Casos Failed precisam de comentários claros.
  • Blocked deve explicar o motivo da impossibilidade.
  • Skipped deve justificar por que o caso não se aplica.
  • Evidências são obrigatórias para Failed e Blocked.
  • Não finalizar a run com casos em branco.
  • Revisar todos os status antes de encerrar.

4. Evidências

Evidências são essenciais para auditoria e rastreamento.

Devem incluir:

  • prints de tela
  • vídeos (quando necessário)
  • logs
  • mensagens de erro

Evidências devem ser anexadas ao resultado, jamais ao case.

5. Comentários Obrigatórios

Casos Failed devem explicar:

  • o que foi feito
  • o que era esperado
  • o que ocorreu de errado

Casos Blocked devem explicar:

  • o motivo da impossibilidade de execução
  • evidências do bloqueio

Casos Skipped devem explicar:

  • por que o teste não se aplica ao cenário

6. Erros Comuns (e proibidos)

  • ❌ Marcar Passed apenas para “terminar rápido”
  • ❌ Marcar Blocked quando na verdade houve falha funcional
  • ❌ Não anexar evidências
  • ❌ Executar caso sem seguir os passos
  • ❌ Não detalhar comentários de falha
  • ❌ Deixar casos sem status
  • ❌ Usar Skipped para mascarar falhas reais

7. Impacto nos KPIs do WellnessQAReporter

Os resultados determinam:

  • taxa de sucesso
  • taxa de falha
  • distribuição por funcionalidade
  • severidade dos defeitos (via Failed)
  • impacto por plataforma
  • consistência da cobertura

Conclusão:

O registro correto dos resultados é crítico para o sucesso do processo de QA e para a precisão dos relatórios do WellnessQAReporter.

A disciplina no preenchimento garante qualidade, rastreabilidade e confiabilidade dos dados.
