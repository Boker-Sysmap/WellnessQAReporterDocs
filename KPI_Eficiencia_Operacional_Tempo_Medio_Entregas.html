KPI – Eficiência Operacional
(Tempo Médio de Entregas)

1. Objetivo do KPI

Este KPI calcula o tempo médio necessário para concluir uma entrega, permitindo medir a eficiência do time de QA e o nível de maturidade do processo.

Ele indica principalmente:

  • Velocidade do time para entregar ciclos completos
  • Agilidade no processo de testes
  • Impacto de gargalos, atrasos e re-trabalho
  • Previsibilidade das releases

2. Regras de Negócio

2.1 RN01 – O que é considerado uma “entrega”?

A entrega pode representar:

  • Conjunto completo do escopo de testes da release
  • Finalização de uma Sprint
  • Finalização de um Plano de Testes
  • Entrega de uma funcionalidade testada

O cliente deve definir qual modelo se aplica ao seu processo — o mais comum é tempo para concluir uma Release.

2.2 RN02 – Marco inicial da entrega

O início do ciclo deve ser definido como:

  • Data de início da Release (mais comum)
  • Data de início do primeiro Test Plan
  • Data da primeira execução

2.3 RN03 – Marco final da entrega

A entrega é considerada concluída quando:

  • 100% do escopo planejado foi executado
  • Todos os defeitos bloqueantes foram resolvidos
  • A Release foi marcada como “finalizada”

2.4 RN04 – Fórmula principal

Tempo médio de entrega = (data_fim – data_início) por ciclo de entrega

Na consolidação:

Tempo médio = soma dos tempos de entrega ÷ quantidade de entregas

2.5 RN05 – Unidade de tempo

  • Dias para Releases e Sprints
  • Horas para grupos menores (ex.: planos de teste)

3. Cálculo do KPI no WellnessQAReporter

3.1 Fonte dos dados

  • /release – datas de início e fim da release
  • /plan – datas dos planos (quando aplicável)
  • /run – progresso real das execuções
  • /result – métricas de execução e finalização

3.2 Algoritmo completo

  1. Identificar cada ciclo de entrega concluído (ex.: releases).
  2. Capturar data de início e data de finalização.
  3. Calcular o tempo de duração:

    tempo = fim − início
  4. Gerar o tempo médio somando todos os ciclos.
  5. Classificar por mês, quarter ou release, conforme desejado.

3.3 Quando a entrega é considerada concluída

O Reporter considera uma delivery concluída quando:

  • Todos os Test Plans vinculados à release foram finalizados
  • Executado 100% do escopo planejado
  • Não há casos “em execução” pendentes

4. Configuração necessária no Qase

4.1 Releases bem definidas

Cada release deve possuir:

  • Data de início
  • Data de fim
  • Nome padronizado

4.2 Test Plans vinculados corretamente

O cálculo só funciona se cada plano apontar para sua release.

4.3 Fechamento formal da release

A equipe deve atualizar o status da release para “concluída” ao finalizar todas as execuções.

4.4 Evitar prorrogações artificiais

Se a release não for encerrada oficialmente, o tempo fica inflado.

5. Riscos e pontos de atenção

Risco Impacto Solução
Release sem data de encerramento Tempo calculado incorreto Fechar release ao final do ciclo
Execuções espalhadas fora do ciclo Distorção no tempo real de entrega Manter governança de calendário
Desalinhamento entre times Indicador perde confiabilidade Padronizar workflow de entrega
Test Plans não vinculados KPI não reflete realidade Verificar associação obrigatória

6. Resumo executivo

  • Mede quanto tempo o time leva para entregar um ciclo completo.
  • Baseado em datas reais de início e conclusão das releases.
  • Ajuda a prever prazos e otimizar eficiência.
  • Exige boa governança no uso de releases e test plans.
  • É um indicador crítico para planejamento estratégico.
