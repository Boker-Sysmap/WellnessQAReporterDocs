Processo de Releases Quinzenais

Esta página descreve o processo oficial das releases quinzenais e as atividades obrigatórias no Qase para garantir que o WellnessQAReporter consiga interpretar corretamente o ciclo, calcular KPIs e gerar relatórios consistentes.

Objetivo:

Padronizar o fluxo de trabalho das releases, eliminando divergências entre equipes e garantindo que as execuções e os dados no Qase reflitam exatamente o estado da qualidade do produto.

1. Estrutura do Ciclo Quinzenal

Cada release segue um ciclo de 15 dias, contendo:

  • preparação do escopo
  • revisão de suites e test cases
  • criação dos test plans
  • criação das test runs
  • execução completa
  • registro de defeitos
  • fechamento e validação final

Cada etapa é essencial para a integridade dos dados.

2. Antes da Release (Preparação)

As seguintes ações devem ser concluídas antes do início oficial da execução:

2.1 Revisão das Suites

  • remover funcionalidades obsoletas
  • atualizar passos e resultados esperados
  • garantir que casos tenham tags de plataforma e idioma

2.2 Definição do Escopo

  • validar com PO ou líder do QA o que será testado
  • incluir novas funcionalidades

2.3 Criação do Test Plan

  • criar um Test Plan por plataforma ou um Test Plan geral (dependendo do projeto)
  • adicionar todas as suites relacionadas

Erro proibido:

  • ❌ começar a executar testes sem o Test Plan criado

3. Durante a Release (Execução)

A execução deve seguir normas que garantem rastreabilidade e qualidade:

3.1 Criação das Test Runs

  • criar test run nova para iOS
  • criar test run nova para Android
  • seguir padrão de nomenclatura
  • nunca reaproveitar test runs antigas

3.2 Execução dos Test Cases

  • executar todos os Test Cases
  • seguir passos descritos nos Test Cases
  • registrar status (Passed/Failed/Blocked/Skipped)
  • anexar evidências quando necessário

3.3 Registro de Defeitos

  • abrir defeito para cada Failed
  • vincular defeito ao resultado
  • preencher descrição completa

3.4 Monitoramento Diário

  • revisar progresso das runs
  • garantir que não há casos pendentes
  • validar se defeitos estão completos

4. Após a Release (Fechamento)

O fechamento garante que todos os dados necessários foram registrados.

Checklist obrigatório:

  • Test Runs 100% completas
  • Todos os casos com status
  • Todos os Failed possuem defeito vinculado
  • Blocked e Skipped possuem justificativa
  • Severidades dos defeitos revisadas
  • Duplicidades removidas
  • Test Plan finalizado corretamente

Somente após este processo o Reporter deve ser executado.

5. Boas Práticas do Ciclo Quinzenal

  • Criar o Plan no primeiro dia da release
  • Criar as runs antes de começar a execução
  • Registrar defeitos no mesmo dia da falha
  • Atualizar suites quando funcionalidades mudarem
  • Fazer revisões rápidas diárias das runs
  • Validar status antes de encerrar a execução

6. Erros Comuns (e proibidos)

Esses erros prejudicam diretamente o Reporter:

  • ❌ Executar sem revisar suites
  • ❌ Esquecer de adicionar suites ao Test Plan
  • ❌ Deixar casos sem status
  • ❌ Executar sem registrar evidências
  • ❌ Criar test runs incompletas e abandoná-las
  • ❌ Abrir defeitos sem vínculo
  • ❌ Reaproveitar Test Plans ou Test Runs

7. Impacto no WellnessQAReporter

O ciclo quinzenal impacta:

  • KPIs de escopo
  • KPIs de cobertura
  • KPIs de execução
  • KPIs de falhas
  • Severidade dos defeitos
  • Tendência de qualidade por release
  • Comparação entre plataformas

Se o processo for seguido corretamente, o Reporter entregará informações precisas e executivas.

Conclusão:

Seguir o processo quinzenal garante organização, previsibilidade e clareza.

É o pilar fundamental para que o WellnessQAReporter gere relatórios confiáveis e úteis para as áreas de QA, PO e gestão.
