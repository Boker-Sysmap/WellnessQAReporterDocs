Processo de Releases e Gestão de Ciclos de QA

Esta página descreve o processo oficial de gestão de releases e as atividades obrigatórias no Qase para garantir que os dados estejam organizados, rastreáveis e compatíveis com a geração de relatórios e indicadores do WellnessQAReporter.

Objetivo

Padronizar o fluxo de trabalho de releases, garantindo consistência, rastreabilidade e organização dos dados de QA, mesmo em projetos onde uma release dura mais de uma sprint.

1. Estrutura do Ciclo

Diferentemente da sprint, que possui duração fixa (normalmente 15 dias), uma release pode abranger uma ou mais sprints, dependendo do conteúdo planejado e das necessidades do produto.

Cada release contém:

  • preparação do escopo
  • revisão de suites e test cases
  • criação dos Test Plans
  • criação das Test Runs
  • execução dos testes (podendo atravessar várias sprints)
  • registro de defeitos
  • fechamento da release e consolidação final

O processo deve ser respeitado para garantir a consistência dos dados entre releases e a confiabilidade dos KPIs históricos.

2. Identificação da Release (Nomenclatura Oficial)

Toda release deve possuir um identificador único, que será usado em:

  • nomes dos Test Plans
  • nomes das Test Runs
  • organização das pastas e históricos
  • arquivos de relatórios gerados

Formato do identificador da release:

{VERSÃO}_{AMBIENTE}

Onde:

  • VERSÃO = 3.2.0
  • AMBIENTE = DEV

O identificador da release será criado automaticamente pelo Reporter a partir dos identificadores dos Test Plans

Exemplos:

  • 3.2.1_STAGE
  • 3.1.0_DEV

3. Antes da Release (Preparação)

As seguintes ações devem ser concluídas antes do início oficial da execução:

3.1 Revisão das Suites

  • remover funcionalidades obsoletas
  • atualizar passos
  • garantir tags de plataforma e idioma, quando aplicável

3.2 Definição do Escopo

  • validar escopo com PO/líder de QA
  • incluir funcionalidades da release

3.3 Criação do Test Plan

  • criar Test Plan seguindo o identificador oficial
  • adicionar todas as suites relacionadas

4. Durante a Release

A execução pode atravessar várias sprints. O importante é manter rastreabilidade e disciplina.

4.1 Criação das Test Runs

  • criar Test Run nova por plataforma, vinculada ao Test Plan da release
  • nunca reaproveitar Test Runs antigas

4.2 Execução dos Test Cases

  • seguir os passos descritos
  • registrar status corretamente
  • anexar evidências quando necessário

4.3 Registro de Defeitos

  • abrir defeito para cada caso Failed
  • vincular ao resultado da execução

4.4 Monitoramento Contínuo

  • acompanhar progresso diariamente
  • revisar status pendentes

5. Fechamento da Release: REGRA CRÍTICA

Uma release só pode ser fechada quando:

  • todas as Test Runs estiverem 100% concluídas
  • todos os casos com status registrado
  • todos os Failed tiverem defeito vinculado
  • Severidades e duplicidades revisadas
  • não houver mais execuções pendentes

Após o fechamento:

  • ✔ a release se torna imutável no histórico interno do WellnessQAReporter
  • ✔ qualquer mudança posterior no Qase não será refletida
  • ✔ o snapshot da release é congelado permanentemente

6. Transição para a Próxima Release – NOVA REGRA

A equipe deve seguir as regras abaixo para evitar inconsistências:

6.1 A nova release só pode ser iniciada quando a anterior for fechada

  • isso evita mistura de dados entre releases
  • garante que KPIs não sejam reprocessados indevidamente

6.2 Enquanto uma release estiver aberta

  • casos podem ser atualizados
  • execuções podem ser refeitas
  • tipos manual/automated podem mudar
  • testes podem ser adicionados ou ajustados

6.3 Após fechar a release

  • nenhuma alteração deve ser feita nos dados
  • toda modificação passa a impactar somente a próxima release

6.4 O WellnessQAReporter aplica automaticamente:

  • ❗ congelamento de releases antigas (não podem ser reprocessadas)
  • ❗ atualização permitida somente para a release mais recente
  • ❗ rejeição silenciosa de tentativas de atualização de releases anteriores

Isso garante que dados críticos como cobertura manual x automatizada — que não possuem histórico no Qase — sejam preservados corretamente.

7. Boas Práticas

  • Criar Test Plans no início da release.
  • Criar Test Runs antes de iniciar a execução.
  • Realizar revisões frequentes nas Test Runs.
  • Registrar defeitos no mesmo dia da execução.
  • Respeitar o identificador oficial da release.
  • Comunicar explicitamente o fechamento da release.

8. Impacto na Organização e Relatórios

O processo de releases e seu fechamento adequado garantem:

  • KPIs consistentes por release
  • comparação histórica confiável
  • cobertura planejada x executada por plataforma
  • cálculo correto de cobertura manual/automatizada mesmo após mudanças no Qase
  • preservação de snapshots históricos independentemente de alterações posteriores

Conclusão

O processo atualizado garante que releases possam durar múltiplas sprints sem comprometer rastreabilidade e consistência. Seguir este fluxo é essencial para que o WellnessQAReporter gere indicadores confiáveis e úteis para tomada de decisão.
