Test Plans

Esta página define como os Test Plans (Planos de Teste) devem ser criados e estruturados no Qase para que o WellnessQAReporter consiga identificar corretamente o escopo planejado e os KPIs de cada release quinzenal.

Objetivo:

Padronizar completamente a criação de Test Plans, garantindo que cada ciclo de release seja representado de forma correta, completa e consistente.

1. O que é um Test Plan?

O Test Plan é o artefato do Qase que define o escopo planejado da release.

Ele contém as suites (funcionalidades) que serão testadas durante o ciclo e serve como base para todas as execuções (runs).

Por que isso é importante?

  • O Reporter usa o Test Plan para calcular KPIs de escopo e cobertura.
  • O Test Plan representa o que deveria ser testado na release.
  • Sem Test Plan, não há como medir percentual de execução.

2. Frequência dos Test Plans

É obrigatório criar um Test Plan para cada release quinzenal, sem exceções.

Nunca reutilize um Test Plan antigo.

Cada ciclo deve ter seu próprio artefato, pois:

  • o escopo muda
  • funcionalidades entram e saem
  • regressões precisam ser revistas
  • o Reporter identifica KPIs por release

3. Nomenclatura Oficial

A nomeação deve seguir o padrão:

Release YYYY-MM – Plataforma

Exemplos:

  • • Release 2025-03 – iOS
  • • Release 2025-03 – Android

Para projetos que utilizam o mesmo escopo nas duas plataformas, pode-se usar:

  • Release 2025-03 – Geral

4. Conteúdo Obrigatório de um Test Plan

Ao criar um test plan, ele deve conter:

  • Todas as suites relacionadas à release.
  • Suites revisadas e atualizadas.
  • Funcionalidades novas adicionadas.
  • Funcionalidades removidas do ciclo quando necessário.
  • Descrição do objetivo da release (opcional, mas recomendado).

Por que isso importa?

O WellnessQAReporter lê diretamente os Test Plans para:

  • identificar o escopo total da release
  • calcular cobertura planejada x executada

5. Revisão Antes da Execução

Antes de iniciar qualquer execução, o QA responsável deve confirmar que:

  • Todas as suites foram incluídas.
  • Não há suites obsoletas.
  • Não há duplicidade de suites.
  • Casos desatualizados foram revisados.

6. Erros Comuns (e proibidos)

Os erros abaixo prejudicam gravemente os KPIs:

  • ❌ Test Plan reaproveitado de release anterior
  • ❌ Suites faltando
  • ❌ Suites duplicadas
  • ❌ Test Plans vazios
  • ❌ Test Plans criados após a execução já ter começado
  • ❌ Test Plans com nome informal (ex.: “Regressão Março”)

7. Boas Práticas

  • Criar o Test Plan no início do ciclo de release.
  • Validar escopo com o PO ou líder de QA.
  • Arquivar Test Plans antigos para manter organização.
  • Criar templates de Test Plans, se necessário.

Conclusão:

Test Plans bem estruturados são essenciais para medir o escopo da release e garantir que o WellnessQAReporter calcule corretamente a cobertura, execução e qualidade do ciclo.

Seguir esse padrão é obrigatório para garantir dados consistentes.
