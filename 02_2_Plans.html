Test Plans

Esta página define como os Test Plans e Test Runs devem ser criados no Qase, seguindo regras obrigatórias de nomenclatura e organização. Essas regras garantem rastreabilidade, organização e consistência entre projetos, versões e plataformas.

Objetivo

Garantir que todo Test Run seja baseado em um Test Plan, e que ambos sigam a nomenclatura padronizada definida oficialmente para o projeto.

1. O que é um Test Plan?

O Test Plan define o escopo planejado de testes para uma versão específica do produto. Ele contém todas as suites (funcionalidades) e deve ser criado antes de qualquer execução.

Por que isso é obrigatório?

  • O Test Plan é a única fonte que define o escopo planejado.
  • Sem Test Plan, não é possível calcular KPIs críticos, como Escopo Planejado.
  • Permite rastrear exatamente o que deveria ter sido testado.

2. Regras Obrigatórias

2.1 Não é permitido criar Test Run sem Test Plan

Todo Test Run deve obrigatoriamente ser criado a partir de um Test Plan.

2.2 Não é permitido reaproveitar Test Plans antigos

Cada Test Run deve possuir um Test Plan próprio, com o mesmo nome (identificador).

Permitido: clonar modelos ou Test Plans anteriores.

Proibido: reaproveitar diretamente um Test Plan já usado em outra execução.

2.3 Test Plan e Test Run devem ter o mesmo identificador

O nome (identificador) deve ser idêntico entre o Test Plan e o Test Run que o utiliza.

3. Padrão Oficial de Identificador

O identificador deve seguir exatamente este formato:

${version}_${environment}_${platform}_${language}_${testType}_${date}

Onde:

  • version — sempre no formato X.Y.Z (ex.: 3.2.1)
  • environment — PROD, STAGE, DEV
  • platform — ANDROID, IOS
  • language — PT, EN, ES
  • testType — FUNCIONAL, MANUAL, REGRESSÃO, AUTOMATIZADO etc.
  • date — data da execução no formato AAAAMMDD
  • Usar somente caracteres MAIÚSCULOS.

Exemplos válidos:

1.2.3_PROD_IOS_PT_[FUNCIONAL_MANUAL]_20251117

1.2.3_PROD_IOS_EN_MANUAL_20251011

3.1 Campos com caracteres especiais

Qualquer campo que contenha espaços, acentos, hífens, barras ou underscores deve obrigatoriamente estar entre colchetes:

1.2.3_prod_IOS_PT_[FUNCIONAL_MANUAL]_20251117

Exemplos de campos que exigem colchetes:

  • funcional manual → [FUNCIONAL MANUAL]
  • regressão completa → [REGRESSÃO COMPLETA]
  • mobile_ios → [MOBILE_IOS]

4. Conteúdo obrigatório de um Test Plan

  • Todas as suites da versão
  • Funcionalidades novas incluídas
  • Funcionalidades removidas quando necessário
  • Test cases revisados antes da execução

5. Erros Comuns (e proibidos)

  • ❌ Criar Test Run sem Test Plan
  • ❌ Reaproveitar Test Plans antigos
  • ❌ Usar nomes de identificador fora do padrão
  • ❌ Esquecer a data AAAAMMDD
  • ❌ Criar Test Plan após iniciar execução

6. Conclusão

A aplicação rigorosa deste padrão é essencial para manter organização no Qase, gerar KPIs corretos e permitir que o WellnessQAReporter interprete os dados de forma consistente.
