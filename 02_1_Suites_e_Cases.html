Suites & Cases

Esta página detalha como as Suites e os Test Cases devem ser estruturados no Qase para garantir que o WellnessQAReporter consiga interpretar corretamente as funcionalidades, plataformas, idiomas e impacto de cada área do aplicativo.

Objetivo:

Estabelecer um padrão claro e obrigatório para criação e manutenção de Suites e Test Cases, evitando duplicações, inconsistências e lacunas de informação.

1. Estrutura de Suites

As Suites devem representar agrupamentos funcionais do aplicativo. Cada suite deve corresponder a uma funcionalidade ou módulo específico.

Padrões obrigatórios:

  • Cada suite deve cobrir uma única funcionalidade.
  • Nomenclatura obrigatória: “Nome da Funcionalidade – Subfuncionalidade (se aplicável)”.
  • Suites genéricas não são permitidas (ex.: “Testes Gerais”, “Diversos”).
  • Suites devem ser revisadas a cada release quinzenal.

Exemplos de nomenclatura:

  • Autenticação – Login
  • Autenticação – Recuperar Senha
  • Pagamentos – Cartão de Crédito
  • Perfil – Edição de Dados
  • Configurações – Idioma

Por que isso é importante?

O WellnessQAReporter gera o “Resumo Funcional” usando as suites para agrupar funcionalidades.

Se as suites estiverem desorganizadas, os KPIs por módulo ficarão incorretos ou incompletos.

2. Estrutura de Cases

Cada caso de teste deve ser completo, objetivo e conter todas as informações necessárias para a execução, auditoria e rastreamento.

Itens obrigatórios de um Test Case:

  • Título claro e padronizado.
  • Descrição do cenário.
  • Pré-condições.
  • Passo a passo detalhado.
  • Resultado esperado.
  • Tags de plataforma (iOS e/ou Android).
  • Tags de idioma (PT/EN/ES), quando aplicável.
  • Severidade do caso (opcional, mas recomendado para regressões).

Padrões de Título:

  • Sempre iniciar com o objetivo do teste.
  • Evitar títulos vagos como “Validar tela A”.

Exemplos corretos:

  • Validar login com credenciais válidas.
  • Validar mensagem de erro ao inserir senha incorreta.
  • Validar seleção de idioma para espanhol.

Exemplos incorretos:

  • Teste login.
  • Teste senha.
  • Verificar tela X.

Por que isso é importante?

O Reporter depende dos títulos para classificações e para documentar o fluxo de teste.

Cases mal descritos prejudicam análises e dificultam rastreamento.

3. Tags

As tags permitem que o Reporter divida corretamente os KPIs por:

  • plataforma (iOS / Android)
  • idioma (PT / EN / ES)
  • categoria funcional (opcional, mas recomendado)

Tags obrigatórias:

  • iOS
  • Android

Tags opcionais (quando aplicável):

  • PT
  • EN
  • ES

Erro comum:

Test Cases duplicados apenas para trocar idioma — não é necessário criar três casos, apenas diferenciar via tags.

4. Boas práticas gerais

  • Nunca duplicar test cases sem motivo.
  • Atualizar test cases quando comportamento do app muda.
  • Usar linguagem clara e objetiva.
  • Evitar siglas não oficiais.
  • Centralizar anexos referentes ao teste apenas nos resultados, nunca no test case.

Conclusão:

Suites e Test Cases bem estruturados garantem rastreamento claro e KPIs confiáveis.

Essas regras são obrigatórias para garantir que o WellnessQAReporter funcione com precisão.
