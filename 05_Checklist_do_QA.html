6 - Checklist do QA

Esta página apresenta o checklist oficial que todos os analistas de QA devem seguir ao trabalhar

com o Qase durante uma release quinzenal. Ele garante padronização, reduz erros operacionais

e assegura que o WellnessQAReporter receba dados completos e consistentes.

Objetivo:

Criar um guia simples, direto e obrigatório que assegure conformidade ao processo de QA e qualidade das informações registradas.

1. Checklist Antes da Execução

Antes de iniciar qualquer run:

[ ] Suites revisadas e atualizadas

[ ] Casos de teste completos (passos, pré-condições, resultado esperado)

[ ] Tags de plataforma aplicadas (iOS / Android)

[ ] Tags de idioma aplicadas (PT / EN / ES), quando aplicável

[ ] Plan da release criado corretamente

[ ] Todas as suites adicionadas ao Plan

[ ] Run criada para iOS

[ ] Run criada para Android

[ ] Nomenclatura correta das runs

[ ] Ambiente validado e funcional

2. Checklist Durante a Execução

Durante a execução dos casos de teste:

[ ] Seguir exatamente os passos do case

[ ] Registrar status de cada caso

[ ] Inserir comentários nos casos não aprovados

[ ] Anexar evidências (prints, vídeos) para Failed e Blocked

[ ] Abrir defeitos imediatamente ao encontrar uma falha

[ ] Garantir vínculo entre defeito e resultado da execução

[ ] Justificar Blocked e Skipped de forma clara

[ ] Atualizar severidade dos defeitos conforme impacto real

3. Checklist Após a Execução

Antes de encerrar a run:

[ ] 100% dos casos com status

[ ] Todos os Failed com defeitos vinculados

[ ] Todas as evidências anexadas

[ ] Blocked e Skipped devidamente justificados

[ ] Sem casos deixados “para depois”

[ ] Revisão completa feita pelo QA responsável

[ ] Duplicidade de defeitos verificada e corrigida

[ ] Severidades revisadas pelo QA líder

[ ] Run encerrada após validação final

4. Checklist Antes de Gerar o Relatório

Para garantir KPIs corretos:

[ ] Plan completo e atualizado

[ ] Runs concluídas para ambas as plataformas

[ ] Nenhum caso sem status

[ ] Nenhum defeito sem vínculo

[ ] Nenhum defeito sem descrição completa

[ ] Suites e cases consistentes com o escopo da release

5. Erros Comuns que o Checklist Evita

❌ Executar runs com cases em branco

❌ Esquecer evidências de falhas

❌ Criar runs fora do padrão

❌ Deixar defeitos sem vínculo

❌ Marcar Passed sem validar comportamento

❌ Revisões tardias que forçam retrabalho

❌ Confusão entre plataformas e idiomas

6. Benefícios do Checklist

• Padroniza a atuação dos QAs

• Reduz inconsistências entre projetos

• Melhora rastreabilidade dos dados

• Elimina erros simples que comprometem KPIs

• Aumenta confiabilidade dos relatórios executivos

• Sustenta a maturidade do processo de QA

Conclusão:

O checklist é obrigatório. Ele assegura disciplina e consistência no processo de QA, garantindo

que o WellnessQAReporter opere com dados confiáveis e gere análises precisas para o time e gestão.
